# AgentDock 完整实施计划

> 本文描述 MVP 完成后的长期路线。20～30 天的首月目标以 `MVP_IMPLEMENTATION_PLAN.md` 为准，完整计划不要求在一个月内全部完成。

## 一、最终目标

AgentDock 负责让 Coding Agent 在受控环境中安全、稳定地执行；CCWhat 负责记录、回放、诊断和评测。两者与离线评测集共同形成：

```text
任务准备 → 安全执行 → 全程记录 → 离线评测 → 问题诊断 → 调整改进 → 重新验证
```

最终系统应支持单个任务、批量任务、多机器执行和多 Agent 协作，但这些能力必须逐步建设。

## 二、阶段路线

### 阶段 0：统一基础协议

- 统一任务、运行、Agent、沙箱和评测编号。
- 定义任务输入、运行事件、产物和评测结果格式。
- 明确 AgentDock 与 CCWhat 的职责边界。
- 建立最小的兼容规则，避免两个项目各自修改后无法联动。

完成标志：一个任务从提交到评测的所有数据都可以正确关联。

### 阶段 1：完成单机 MVP

- 在独立 Workspace 中运行 Coding Agent。
- 支持日志、超时、取消、文件变化和结果保存。
- 接入离线评测集与 CCWhat。
- 支持回放、重新运行和两次结果对比。

完成标志：完整演示“执行—观测—评测—改进—重跑”闭环。

### 阶段 2：加强单机执行能力

- 确保任务结束后完整回收所有进程。
- 限制 CPU、内存、进程数量、运行时间和输出量。
- 加强文件与网络访问控制。
- 固定运行环境，提高重复执行的一致性。
- 建立异常任务、安全测试和压力测试集合。

完成标志：失控或恶意任务不能影响宿主机和其他任务，并且所有失败都有证据可查。

### 阶段 3：增加任务控制平台

- 使用 Go 提供任务提交和查询接口。
- 增加任务队列和多个 Worker。
- 支持 Worker 注册、健康检查和任务占用期限。
- 支持取消、有限重试和 Worker 故障后的重新分配。
- 将任务与每次执行尝试分开记录。
- 增加基础并发额度和租户配额。

完成标志：多台 Worker 可以共同处理任务，其中一台退出后任务仍能继续完成。

### 阶段 4：补齐安全与恢复

- 每个任务使用短期且权限受限的身份凭证。
- 安全地向任务提供必要密钥，并记录访问行为。
- 定期保存 Workspace、Agent 状态和运行进度。
- 长任务中断后从最近进度恢复。
- 隔离不同用户的文件、凭证、任务和运行记录。
- 让 CCWhat 展示故障、恢复和任务迁移过程。

完成标志：能够演示任务中断恢复，并证明不同用户之间不会互相访问数据。

### 阶段 5：完善服务和使用体验

- 提供稳定的命令行工具和 Python SDK。
- 提供 Codex、Claude Code 和自定义 Agent 接入方式。
- 支持批量评测、结果导出和运行环境模板。
- 提供清晰错误信息、版本管理和升级说明。
- 建立完整 Quick Start、示例项目和部署文档。

完成标志：不了解内部实现的用户也能快速接入自己的 Agent。

### 阶段 6：多 Agent 协作

- 将 Planner、Executor、Reviewer 和 Tester 作为独立任务运行。
- 为不同角色配置不同的文件、命令和网络权限。
- 角色之间只通过明确的产物传递信息。
- 支持单个节点重试、取消、回放和评分。
- 在 CCWhat 中展示整条协作链路。

完成标志：一个多 Agent 任务可以逐节点执行、失败恢复和完整回放。

### 阶段 7：更多运行环境与规模化

- 抽象统一的运行环境接口。
- 根据实际需求增加容器或 Kubernetes 后端。
- 优化大量并发任务下的调度和资源利用率。
- 增加公平调度、优先级和更细的租户限制。
- 建立长期运行的稳定性与性能报告。

完成标志：系统可以在多机器环境稳定运行，并用数据说明容量和瓶颈。

## 三、贯穿所有阶段的工作

每个阶段都必须同时完成：

- 自动测试。
- 失败案例。
- 性能数据。
- 安全边界说明。
- 架构和使用文档。
- 与 CCWhat 的数据兼容验证。

不要等功能全部完成后再补测试和文档。

## 四、优先级

### 必须做深

- 单机安全执行。
- 超时、取消和完整进程回收。
- 任务重新执行和结果复现。
- CCWhat 观测、回放和评测闭环。
- Worker 故障恢复。
- 安全测试和真实性能数据。

### 后期再做

- Kubernetes。
- MicroVM。
- 复杂的跨地域调度。
- 大规模多 Agent DAG。
- 自动 Prompt 优化。
- 自研消息队列和底层容器技术。

## 五、最终交付形态

完整项目最终应包含：

- Rust Sandbox Worker。
- Go Control Plane。
- Python SDK 和 Agent Adapter。
- CCWhat 可观测与评测集成。
- 安全模型、运行协议和恢复方案文档。
- 安全、故障和性能测试集。
- 单 Agent 与多 Agent 示例。
- 可重复执行的演示和性能报告。

最终评价标准不是功能数量，而是：

```text
能够真实执行
> 安全边界明确
> 失败可以恢复
> 过程可以观测
> 结果可以评测和复现
> 系统可以逐步扩展
```
